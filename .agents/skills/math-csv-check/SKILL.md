# `math-csv-check` スキル

このスキルは、数学問題データ（CSV形式）の品質を維持するために使用します。数学的・形式的な表現の誤りを修正し、整合性を検証します。

## ワークフロー

### 1. 自動バリデーションスクリプトの実行
まず、既存のスクリプトを実行して基本的な形式チェックを行います。

```bash
python .agents/skills/pdf-to-math-csv/scripts/validate_csv.py <対象CSVパス>
```

このスクリプトは以下を自動修正・警告します：
- 先頭ゼロの除去（例: `07` → `7`）
- `options` が正しいJSON配列（4要素）かチェック・修正
- `answer_index` が 1〜4 の整数かチェック
- LaTeXデリミタ `\( ... \)` の開閉不一致を警告

### 2. 詳細バリデーション項目（精読チェック）
AIエージェントは以下のカテゴリに従って1行ずつ厳格に点検を実施してください。

#### A~C. 数値・分数表記ルール
- **数値表現**: 先頭の不要なゼロ（`01`→`1`）や、結果が整数になる場合の末尾ゼロ（`8.0`→`8`）を修正。`\frac{0}{n}` は `0` に、`\frac{n}{1}` は `n` に修正。
- **分数ルール**: 
    - 必ず既約分数にする（`\frac{6}{9}` → `\frac{2}{3}`）。
    - マイナス符号は分母・分子ではなく分数の前に置く（`-\frac{a}{b}`）。
    - 分子・分母の両方にマイナスがある場合はプラスにする。
    - 分子＝分母の場合は `1` に修正。
- **問題文中の分数**: 値が1になる分数は `1` に修正。ただし、通分して提示している計算式中の分数は変更しない。

#### D. LaTeX・記号・記法
- **アポストロフィ混入**: 数式内に混入した `'`（OCRミス等）を除去。
- **演算記号**: `×` ではなく `\times` を、`＝` などの全角記号は半角に統一。
- **LaTeXエスケープルール (重要)**:
    - `question_text`, `explanation`: プレーンテキスト。バックスラッシュは1本（`\(`）。
    - `options`: JSON配列文字列。バックスラッシュは2本（`\\(`）。

#### E. 選択肢 (options) と代数的同値性
- **重複排除**: 数学的に同じ値の別表記（例: `\frac{-3}{2}` と `\frac{3}{-2}`）が選択肢に含まれる場合は、不正解側を別の妥当な誤答（計算ミス等）に差し替える。
- **代数的同値チェック (E4)**: 中学数学の範囲での同値パターンを確認し、正解と同じ値の選択肢を排除する。
    - **累乗の展開**: `(ab)^2 = a^2b^2`
    - **指数法則**: `a^2 \times a^3 = a^5`
    - **分配法則・因数分解**: `2(x+3) = 2x+6`, `x^2-1 = (x+1)(x-1)`
    - **係数1の省略**: `1 \times x = x`

#### F~I. 整合性と論理
- **answer_index**: 1始まりであることを確認し、実際に問題を解き直して正解を指しているか検証。
- **数学的正誤検証 (I)**: AIが実際に計算して `answer_index` が指す選択肢が正しいか検証。不一致の場合は**自動修正せず人間に確認を求める**。
- **解説 (explanation)**: 解説文の中間計算が正しいか、表記ルール（分数の符号等）に違反していないか。

#### J~K. 画像とLaTeX構文
- **画像確認**: `image_url` がある場合、画像内容と問題文の整合性を確認。
- **LaTeX構文**: 中括弧 `{}` の対応、空のデリミタ `\( \)`、全角文字の混入を修正。K1〜K8のチェック項目を遵守。

### 3. レポートと修正
修正を行った場合は以下の形式で報告してください。

```markdown
### 【修正レポート】ファイル名: xxx.csv
- **行X [チェックID]**: (理由)
  - 修正前: ...
  - 修正後: ...
---
### 【要確認】 (自動修正せず人間に確認が必要なもの)
- **行N [I]**: answer_index が数学的に不一致の疑い。計算結果は A。
```

## 注意事項
- `answer_index` は **1始まり**（1〜4）。
- 文字化け防止のため、数式は必ず `\( ... \)` で囲む（デリミタなしは非推奨）。
- 解説文に AI の独白（「すみません」「計算し直します」等）を絶対に含まない。

## 追記: 記述式イベント問題への対応

このスキルは、従来の4択問題CSVを壊さずに、Formix の記述式イベント問題も扱います。
記述式は「1イベント1問」「1人最大3回提出」「スコアは総合ランキングに含めない」「得点に応じてXPのみ付与」という前提です。

### 入力からCSVへ整形する手順

ユーザーから「問題・答え・解説」を受け取ったら、記述式イベント用CSVへ整形します。
未指定の場合は、`question_type=written`、`options=[]`、`answer_index=1`、`written_attempt_limit=1`、`event_status=active` とします。

記述式CSVヘッダー:

```csv
unit_id,category,question_text,options,answer_index,explanation,image_url,question_type,model_answer,grading_rubric,written_attempt_limit,event_status,event_starts_at,event_ends_at
```

列の扱い:

- `question_type`: 記述式は必ず `written`。4択は空欄または `multiple_choice`。
- `options`: 記述式では `[]`。
- `answer_index`: 記述式では `1`。
- `explanation`: 生徒に返してよい短い解説。元資料の逐語コピーは禁止。
- `model_answer`: Gemini採点に渡す模範解答。途中式、最終答、許容表記、部分点条件を含める。
- `grading_rubric`: Gemini採点に渡す採点観点。CSV内ではJSON配列文字列にする。
- `written_attempt_limit`: 原則 `1`。
- `event_status`: `draft`, `active`, `closed` のいずれか。
- `event_starts_at`, `event_ends_at`: ISO 8601形式推奨。例 `2026-06-01T00:00:00+09:00`。

### Gemini採点用データの作り方

`model_answer` には、正しい途中式の代表例、最終答、同値な答え方、表記ゆれ、単位の扱い、部分点条件、減点すべき典型ミスを含めます。
`grading_rubric` は、合計100点になるように設計します。原則は **過程・記述60点、最終答・結論40点** です。

採点API側では、答案画像から読み取った内容を `transcription` として返します。問題作成側の `model_answer` や `grading_rubric` には、OCR・文字起こし・読めない箇所の扱いなどの汎用指示を入れないでください。ここでは問題固有の数学的採点基準だけを記述します。読み取り精度に関する指示は Cloud Functions の共通採点プロンプトに集約します。

採点上限の共通ルール:

- 最終答・結論が数学的に誤っている場合、どれだけ途中が良くても原則60点を上限にする。
- 最終答だけ正しく、立式・説明・証明・途中計算がほぼない場合、原則40点を上限にする。
- 最終答が正しく、過程が一部不足している場合は、40点に過程点の妥当な分だけを加える。
- 最終答がない、問いに対応していない、何を求めたか曖昧な場合、原則60点を上限にする。
- 偶然正答に戻っただけの矛盾した答案や、根拠のない公式使用は大きく減点する。
- 採点者が行間を補って好意的に解釈しない。答案に書かれている内容だけで採点できるルーブリックにする。

例:

```json
[
  "過程・記述: 60点 - 条件設定、立式、途中計算、根拠の説明が答案から読み取れる",
  "最終答・結論: 40点 - 問いに対応した正しい答えを、必要な単位や対象を含めて示している"
]
```

#### 厳格採点用ルーブリック設計

記述式の `grading_rubric` は、最終答が合っているだけで高得点になりすぎないように設計します。原則として、次の観点を含めてください。

- **条件設定・変数定義: 10〜15点** - 文字を使う問題では「もとの半径を \(r\)、高さを \(h\) とする」など、使用する文字の意味を明示しているかを採点する。
- **公式・立式・方針: 20〜30点** - 問題条件を式や証明方針に正しく反映できているか。公式の誤用、比較対象の取り違え、条件の読み落としは大きく減点する。
- **途中計算・論理展開: 20〜30点** - 変形、約分、場合分け、証明の根拠などが筋道立っているか。途中式の飛躍が大きい場合は満点にしない。
- **表現の明確さ: 5〜10点** - 記号の意味、単位、言葉での説明、答案の読みやすさ。採点対象外にしない。
- **最終答・結論: 40点** - 数値、式、証明すべき結論などが正しく、問いに対応しているか。値だけでなく「もとの体積の \( \frac{3}{4} \) 倍」のように対象まで示す。

特に、文字式を用いる問題では「未定義の文字を突然使う」「最後に何の何倍かを書かない」「計算過程がなく答えだけを書く」答案を明確に減点できるようにしてください。

問題タイプ別の配点テンプレート:

- **計算・方程式型**: 過程60点 = 方針/公式20、途中計算30、表記10。結論40点 = 正しい値・形・単位。
- **文章題・立式型**: 過程60点 = 条件整理/変数定義15、立式25、計算/比較15、表現5。結論40点 = 問いに対応した答え。
- **証明型**: 過程60点 = 仮定/条件整理15、根拠の明示20、論理の接続20、表現5。結論40点 = 証明すべき主張が正しく導かれている。
- **図形・関数の説明型**: 過程60点 = 図形/関数条件の読み取り15、定理/性質の適用20、計算または推論20、表現5。結論40点 = 求める量や性質を正しく示す。

### 著作権・表現調整ルール

ユーザーが問題・答え・解説を渡した場合でも、解説文はそのまま文字起こししないでください。
教材、問題集、PDF、Webページ由来の解説は、次の方針で再構成します。

- 元の説明順序や言い回しをそのまま残さず、学習者向けに短く書き換える。
- 途中式は数学的に必要な範囲で示すが、解説本文の表現は独自にする。
- 「なぜその操作をするか」を中心にし、元資料の文体・比喩・例示をコピーしない。
- どうしても引用が必要な場合は短い断片に留め、原則は要約・再説明にする。

### 記述式のチェック項目

1. `question_type` が `written` である。
2. 同じ `unit_id` の記述式イベントに問題が複数入っていない。
3. `options` が `[]` で、`answer_index` が `1`。
4. `model_answer` が空でない。
5. `grading_rubric` がJSON配列で、採点観点の合計が概ね100点になる。
6. `explanation` が元資料の逐語コピーになっていない。
7. `written_attempt_limit` が1、または明示的なイベント設計上の例外として1〜3の範囲内。
8. `event_status` と開催期間が意図通り。

### 検証

記述式CSVも既存の検証スクリプトで確認できます。

```bash
python .agents/skills/pdf-to-math-csv/scripts/validate_csv.py <対象CSVパス>
```

`question_type=written` の行では、4択の「選択肢4つ」ルールを外し、`model_answer` と `grading_rubric` の存在・形式を確認します。
